E-commerce ETL

Стек: Python, PostgreSQL, ClickHouse, onETL, PySpark, FastAPI, Celery, Redis, Docker, Kubernetes

Собрал ETL-пайплайн для данных интернет-магазина и выстроил полный маршрут данных от источника до аналитической витрины. Проект загружает данные в PostgreSQL, читает их с помощью onETL и Spark, обрабатывает в Spark и записывает результат в ClickHouse.

  1. Разделил пайплайн на полный и инкрементальный проход, вынес оба режима в отдельные Spark job и добавил HWM в YAML-файл. Это позволило отдельно запускать первый full snapshot, повторно прогонять весь набор данных и обрабатывать только новые данные после последней метки.
  2. Собрал единые настройки сервиса через config.yaml, .env, ConfigMap и Secret, чтобы один и тот же сервис запускался локально и в Kubernetes без ручной правки параметров перед каждым стартом.
  3. Поднял HTTP API на FastAPI и фоновые задачи на Celery с Redis. Через API можно запускать full и incremental ETL, смотреть статус конкретной задачи и хранить историю последних прогонов.
  4. Доработал слой обработки, добавил фильтры для бот-активности и вынес чтение источника, трансформации и историю задач в отдельные сервисы, чтобы пайплайн было проще развивать и тестировать.
  5. Подготовил два сценария запуска локально через Docker Compose и в Kubernetes через Dockerfile, spark-submit job и набор манифестов для Minikube.

По итогам проекта мне стало понятнее, как собирать ETL-сервис с двумя режимами загрузки, держать конфигурацию в одном месте и отдельно выстраивать локальный и Kubernetes-сценарии запуска.

Posts Data Vault

Стек: Python, PostgreSQL, Docker Compose, Data Vault 2.0

Собрал ELT-проект, который забирает данные из REST API, загружает их в слой STG в PostgreSQL и перекладывает в DDS по модели Data Vault 2.0.

  1. Поднял локальный PostgreSQL через Docker Compose и подготовил .env для локального запуска, чтобы Python-скрипты и база работали в одном окружении.
  2. Создал схемы и таблицы для слоев STG и DDS, чтобы отдельно хранить исходные строки, бизнес-ключи, связи и описательные атрибуты.
  3. Написал первый ELT-скрипт, который забирает данные из REST API, очищает таблицу stg.posts и загружает в нее свежий снимок источника вместе с техническими полями.
  4. Написал второй ELT-скрипт, который читает STG, считает hash key и загружает данные в hub, link и satellite-таблицы слоя DDS.
  5. Разложил id, user_id, title и body по модели Data Vault 2.0 так, чтобы повторный запуск не дублировал одинаковые версии деталей поста и модель оставалась прозрачной по сущностям и связям.

После проекта мне стало понятнее, как собирать простую ELT-цепочку от API до DDS, разделять сущности в Data Vault и не плодить дубли в satellite-таблице.

Flights DWH

Стек: Python, PostgreSQL, Apache Airflow, S3, Jupyter

Собрал свою часть проекта по хранилищу данных для рейсов, где STG-слой принимает исходные файлы, приводит поля к единому виду и загружает данные в PostgreSQL.

  1. Подготовил схемы team_vdga_stg и team_vdga_metadata, таблицы flights_raw и airports, а также технический лог etl_load_log, чтобы первый слой хранилища и метаданные жили отдельно.
  2. Настроил загрузку справочника аэропортов из файла ourairports.csv в PostgreSQL, чтобы справочные данные попадали в STG вместе с основным потоком.
  3. Собрал загрузчик рейсов из S3 в формате csv.gz и сделал его инкрементальным, чтобы в STG попадали новые данные без повторной полной перезаливки.
  4. Подключил шаги загрузки к Airflow, чтобы STG-слой запускался как отдельной частью пайплайна и одновременно встраивался в общий проход по слоям.
  5. Проверил результат через SQL и Jupyter: данные появились в STG, ключевые поля не пустые, а лог загрузки совпадает с фактическими строками в таблицах.

Этот проект помог мне лучше понять, как устроен первый слой DWH, зачем ему нужен технический лог и как подготовить STG так, чтобы следующие слои уже не работали с сырыми файлами.
